## 问题背景

这篇记录的是我在武汉市测绘研究院算法实习期间参与的外立面缺陷检测科研项目中，最先遇到的高分辨率小目标检测问题。

在建筑外立面缺陷检测项目里，我一开始遇到的第一个问题是：原始无人机照片分辨率很高，人眼看缺陷还算清楚，但直接丢给 YOLO 检测时，召回率并不稳定。

这听起来有点反直觉。很多人会觉得图片越清晰，模型越容易检测。但目标检测模型实际看到的不是原图，而是缩放到固定输入尺寸后的图像。对于 4000px、6000px 甚至更高分辨率的外立面照片来说，整图缩放会让原本就很小的裂缝、剥落、露筋等缺陷继续变小。

核心矛盾是：

> 原图很大，不代表模型看到的缺陷也很大。模型真正学习的是输入尺寸下剩余的有效像素。

## 外立面缺陷为什么容易变成小目标

建筑外立面场景里，缺陷目标有几个特点：

- 缺陷面积小：裂缝可能只占整张照片的极小区域。
- 背景复杂：窗框、管线、阴影、纹理、污渍都可能干扰检测。
- 目标形态不规则：裂缝细长，剥落边缘不规则，泛碱边界模糊。
- 拍摄距离不固定：无人机和墙面的距离变化会导致同类缺陷尺度变化很大。
- 图片分辨率高：整面墙被拍进一张图，缺陷在全图中占比进一步降低。

所以这类任务本质上不是普通的“中等尺度目标检测”，而是“大图中的小目标检测”。

## 为什么直接 resize 会出问题

YOLO 系列模型通常会把输入图像缩放到指定尺寸，例如 640、960、1280 等。假设原图是 6000 x 4000，模型输入是 640 x 640，那么图像会被大幅压缩。

如果某个裂缝框在原图中只有 80 x 20 像素，缩放到输入图后可能只剩下不到 10 像素的宽高。对模型来说，这个目标已经接近纹理噪声。

可以用一个简单公式理解：

```text
resizedWidth = defectWidth * modelInputWidth / originalImageWidth
resizedHeight = defectHeight * modelInputHeight / originalImageHeight
```

当 resizedWidth 或 resizedHeight 太小，模型就很难稳定学习到这个目标。

### 缩放风险估算伪代码

```text
function estimateDefectPixels(originalImage, defectBox, modelSize):
  scaleX = modelSize.width / originalImage.width
  scaleY = modelSize.height / originalImage.height

  resizedWidth = defectBox.width * scaleX
  resizedHeight = defectBox.height * scaleY

  if min(resizedWidth, resizedHeight) < 8:
    return "high risk: defect may disappear after resizing"

  if min(resizedWidth, resizedHeight) < 16:
    return "medium risk: detection may be unstable"

  return "ok"
```

这段伪代码不是严格阈值，只是用来说明判断思路：检测难度不只取决于原图清晰度，还取决于目标缩放到模型输入后还剩多少有效像素。

## 训练集和现场图像分布不一致

项目早期还遇到另一个问题：一些公开裂缝数据集在局部图片上效果不错，但放到完整外立面场景里就不稳定。

原因是训练数据和实际推理场景不一致：

| 对比项 | 局部裂缝数据 | 完整外立面航拍图 |
| --- | --- | --- |
| 目标占比 | 大 | 小 |
| 背景复杂度 | 低 | 高 |
| 上下文 | 缺陷局部 | 整面墙、窗户、纹理 |
| 类别边界 | 相对清晰 | 容易和污渍、阴影混淆 |
| 推理方式 | 整图输入 | 大图缩放或切片 |

如果训练数据基本都是局部裁剪图，模型会更像是在做“局部缺陷识别”；而实际系统要做的是“完整外立面中的缺陷发现”。这两个任务的难度不一样。

## 我采用的解决思路

我没有只从“换更大模型”这个方向解决问题，而是把重点放在数据和推理方式上：

1. **先分析目标尺度**
   统计缺陷框在原图中的宽高、面积占比，以及缩放到模型输入后的有效像素。

2. **重构数据集**
   清理标注质量差、类别混乱、拍摄风格差异过大的样本，避免模型学习到数据集偏差。

3. **引入 SAHI 切片**
   把大图切成重叠小图，在切片上检测，再把结果还原到原图。

4. **补充现场场景标注**
   使用 LabelMe 对实际外立面场景补标，让训练数据更接近项目数据。

5. **按场景拆分验证集**
   避免相邻照片同时进入训练集和验证集，导致指标虚高。

> 大图直接缩放会压缩小缺陷；切片后再检测，可以保留更高的有效分辨率。

## 实验观察

在这个项目里，我对比过几种思路：

- 直接整图 resize 推理：速度快，但小缺陷容易漏检。
- 提高输入尺寸：有一定改善，但显存和推理耗时压力变大。
- 局部裁剪后检测：缺陷更清楚，但需要解决裁剪覆盖和结果合并。
- SAHI 切片推理：综合效果更适合工程系统。

最终我更倾向于 SAHI，不是因为它一定“指标最高”，而是因为它更适合这个工程场景：

- 能保留小缺陷细节。
- 能控制切片大小和重叠比例。
- 能把结果还原到原图坐标。
- 后续可以接入人工复核和报告流程。

## 工程判断

建筑外立面缺陷检测里，不能只问“YOLO 能不能检测”，而要先问：

- 缺陷在原图中有多大？
- 缩放到模型输入后还剩多少像素？
- 训练数据和现场图像是否同分布？
- 是否有足够外立面场景样本？
- 推理结果是否能回到原图和三维模型中？

如果这些问题没有处理好，换模型往往只能带来有限改善。

## 小结

这篇文章的核心结论是：

> 高分辨率建筑外立面图像不能直接等价为“模型看得清”。整图缩放后，小缺陷可能会被压缩到模型难以识别的尺度。

因此，外立面缺陷检测更合理的路线是：先分析目标尺度，再整理数据集，最后结合切片推理和人工复核构建完整流程。

这也是后续使用 SAHI、集成 LabelMe 和做三维联动的基础。
