## 写在前面

这篇不是单一功能笔记，而是我在这段车载终端测试开发实习期间，把做过的测试场景、排查方法和终端命令整理成的一份归档总结。

由于项目有保密要求，下面对部分平台名、字段名和日志细节做了适度脱敏，但测试思路、验证顺序和排查方式是完整的。整体风格我尽量按 CSDN 精品技术文章来写，方便后续自己回看，也方便别人快速理解车载终端测试是怎么做的。

这段实习最核心的工作，可以概括成四句话：

- 验证 TBOX 在定位、通信和广播报文上的稳定性。
- 验证运营平台和车云平台的数据上报链路。
- 验证诊断、ECALL、WiFi、4G/5G 和电源模式切换。
- 把一次性测试经验整理成可复用的检查清单、脚本和命令记录。

缺图占位：这类内容大多是终端、台架、抓包和日志截图，后续可以补少量示意图，但正文即使没有图片也可以成立。

## 一、实习目标

我参与的不是单一脚本测试，而是一整套车载终端功能验证。测试对象主要围绕 TBOX 和相关通信链路展开，关注的是“它能不能稳定工作、异常后能不能恢复、上报链路会不会断、切换场景会不会掉状态”。

测试过程中最常见的问题不是一开始就报错，而是：

- 某个场景第一次能过，切换一下电源状态就不稳了。
- 某个数据页面看起来正常，但平台回传晚了几秒。
- 某个功能单独测没问题，一旦和另一条链路并发就开始异常。
- 某个恢复机制触发了，但恢复后状态没有完全回零。

所以这类工作不能只看“功能有没有”，还要看“状态机是不是完整”“时序是不是稳定”“异常恢复是不是闭环”。

## 二、TBOX 定位与通信功能测试

我前期做得最多的一块，是 TBOX 定位和通信能力验证。这个部分不是简单看定位有没有，而是要把不同运行状态都跑一遍。

### 重点场景

| 场景 | 关注点 | 典型验证方式 |
| --- | --- | --- |
| 休眠唤醒后定位 | 唤醒后能否稳定获取定位 | 观察日志、平台状态和定位结果 |
| 上电后定位 | 冷启动后首次定位是否正常 | 记录首次定位耗时 |
| 长时间连续定位 | 连续运行是否会失效 | 长时间稳定性测试 |
| 550 模组单独重启 | 重启后自身定位和上层链路是否受影响 | 重启后重新验证定位和上报 |
| 辅助定位 | 缓存是否正确使用、文件丢失后是否能重下 | 清缓存、删辅助定位文件后复测 |
| 漂移检测 | 静态和动态轨迹是否一致 | 对比 NMEA、平台轨迹和实际路线 |

这里最容易踩坑的是缓存。休眠前如果缓存没有清掉，唤醒后可能会优先使用过期数据，导致时间回退或者定位结果不干净。所以我在验证时会先做一次缓存清理，再看首次定位时长和定位质量是否符合预期。

### 常见关注点

- 唤醒后是否能快速恢复定位。
- 上电后首帧报文是否按时发出。
- 长时间运行后是否出现定位漂移或丢失。
- 模组单独重启后，是否影响其它链路。
- 辅助定位文件缺失后，是否能自动重新下载。

## 三、车载运营平台数据上报机制

这个部分更像“系统联动测试”，因为不仅要看终端本地，还要看平台侧是否能正确接收和展示。

我整理下来，平台侧最关心四类数据：

1. 定位数据
2. SOS 信息
3. 质检结果
4. 故障数据

其中定位和 SOS 通常需要双备份上报；质检和故障更多依赖固定前提条件，比如车辆是否处于 ON 电状态、某些模块是否允许检测等。

### 重点检查项

- 在线定位是否按周期上报。
- 离线定位是否能按定时唤醒周期上报。
- ON 电跳转 OFF 电前，是否会补发一次定位。
- SOS 触发后，VIN、颜色、电池等信息是否完整。
- 质检和故障上报是否受状态前提限制。
- 配置下发时，运营平台和本地配置谁优先。

### 我怎么理解这部分

这类链路不是单纯“能发出去就行”，而是要满足：

- 发得准：字段别错。
- 发得全：关键字段别丢。
- 发得稳：网络抖动、平台切换、重连后还能继续。
- 发得回：失败后要能补发，不能悄悄丢掉。

如果把它抽象成一条链路，就是“终端采集 -> 本地状态判断 -> 平台交互 -> 结果回写 -> 异常重试”。

## 四、诊断、代理应用、ECALL 和网络切换

后面我还处理过一组偏联动的测试：远控/诊断代理、DOIP、ECALL、WiFi、4G/5G 和 APN。

### 典型场景

- 平台代理能不能正常连上。
- 代理应用有没有返回结果。
- 返回失败时，是否支持补发。
- WiFi 休眠唤醒或重启后能否自动恢复连接。
- DOIP 会话切换是否正确。
- ECALL 白名单、使能开关和拨打流程是否正确。
- 4G / 5G 切换、内外天线切换和双 APN 是否符合预期。

这部分最有代表性的感受是：表面上看是几个独立功能，实际上底层共用很多状态和链路。比如网络不好时，平台代理、远程诊断、SOS、定位和状态上报都会一起受影响。

### DOIP 和终端会话

诊断相关测试里，我会重点确认会话切换是否正确，以及一些响应码是否符合预期。这里更像是在验证“终端有没有按标准说话”，而不是只看一个界面按钮。

### ECALL

ECALL 的验证重点不是“能不能拨出去”这么简单，而是：

- 白名单电话是否配置正确。
- 使能开关是否打开。
- 呼入、呼出、重播和静音逻辑是否正确。
- 故障检测和指示灯状态是否一致。

## 五、电源模式与唤醒逻辑

这部分是整车状态机里很重要的一环。

- 仓储、休眠、正常三种模式要区分清楚。
- 不同唤醒方式对应的行为也不一样。
- 预约唤醒和定时唤醒的差异要特别注意。
- 短时中断和主电断开时，终端要不要继续发报文，也要单独验证。

这个模块的价值在于，它决定了很多功能是在什么状态下能不能工作。很多表面看起来像通信问题的故障，最后其实是电源模式切换时的状态没有对齐。

## 六、我在终端里常用的命令

这部分没有图片，直接把我经常用的终端命令整理成速查表。不同环境命令略有差异，下面写的是比较通用的思路。

```bash
# 看系统日志
journalctl -f
dmesg -w

# 看模块和驱动
lsmod
lsmod | grep -E "wifi|can|gps"

# 看网络状态
ip addr
ip route
ping 8.8.8.8

# 看 CAN 总线
candump can0
candump can1

# 发送 CAN 报文（示例）
cansend can0 43B#0102030405060708

# 看进程和资源占用
ps -ef | grep -i tbox
top

# 看无线状态（示例）
nmcli dev status
nmcli dev wifi list
```

这些命令的作用不是“炫技”，而是快速判断问题属于哪一层：

- 系统层：驱动、模块、进程、日志。
- 网络层：IP、路由、DNS、连通性。
- 总线层：CAN / CANFD 报文是否正常。
- 应用层：诊断、定位、ECALL、平台交互是否正常。

## 七、工具链

这段实习里，我实际用到的工具包括：

- MobaXterm
- PCAN
- CANFD
- tensorCRT

如果把工作流说得更直白一点，就是：

1. 先连终端看日志。
2. 再用 CAN 工具看总线报文。
3. 再看平台和终端状态是否一致。
4. 最后再验证异常恢复和补发机制。

## 八、这段实习给我的收获

我以前做功能测试时，更多是在看“有没有”。这段实习让我更强烈地意识到，车载测试真正重要的是：

- 状态前提有没有搞清楚。
- 异常恢复有没有闭环。
- 平台、终端和总线三边的数据有没有对上。
- 一个功能单独没问题，不代表合起来也没问题。

对我来说，这段经历的价值不只是“做过哪些测试”，而是把车载系统从终端、通信、平台和恢复机制四个层面串起来了。后面如果再写项目总结或面试复盘，这些内容都可以直接拿来用。
