## 写在前面

这篇继续拆分测试内容，重点看“终端之外”的那条链路：平台上报、诊断联动、ECALL、网络切换和配置下发。

如果说前一篇是测 TBOX 能不能自己跑稳，这一篇就是测它和外部系统能不能对齐。

## 一、平台上报先看什么

平台上报里，我重点看四类数据：

- 定位
- SOS
- 质检
- 故障

这里的核心不是“有没有发”，而是：

- 发的数据是不是完整。
- 发的时机是不是正确。
- 失败后能不能补发。
- 平台和终端的状态是不是一致。

### 在线 / 离线定位

在线定位是在车辆正常工作时周期上报。

离线定位是在休眠状态下按策略唤醒后上报。这个场景里最重要的是两个判断：

1. 唤醒策略有没有生效。
2. 超时后是怎么收尾的。

如果超时了，系统不能一直卡着，要么上空定位，要么进入休眠，至少流程要闭合。

### SOS

SOS 这种链路更像“事件型上报”。

我会重点确认：

- 触发源是否正确。
- 关键字段是否完整。
- 误触发和挂断逻辑是否正确。
- 本地缓存是否能跨休眠保存。

## 二、诊断联动怎么测

诊断这块我主要关注会话切换、响应码和复位行为。

可以把它理解为：终端有没有按标准在“说话”。

我习惯先确认三件事：

```text
默认会话是否正常
扩展会话是否正常
编程会话是否正常
```

如果会话切换有问题，后面的诊断服务、复位、故障读取都会跟着异常。

### 常看的诊断动作

- 会话切换。
- 软复位 / 硬复位。
- 故障码读取。
- 诊断应用重启后的状态恢复。

## 三、ECALL

ECALL 的测试重点不只在拨打成功，而在于完整链路：

- 白名单是否配置好。
- 使能是否打开。
- 触发方式是否正确。
- 指示灯状态是否和功能状态一致。
- 失败后是否有重拨或重试机制。

这类功能很典型：单点看没问题，和网络、音频、状态灯一起联动后才容易暴露真实问题。

## 四、网络和配置

平台相关功能经常会受网络影响，所以我也会同时看：

- WiFi 重连。
- 4G / 5G 切换。
- 内外天线切换。
- 双 APN 的可用性。
- 平台代理是否能正常返回结果。

另外，配置下发时也要确认优先级：

- 平台下发的配置是不是高优先级。
- 本地配置会不会被覆盖。
- 平台返回空配置时，终端是不是保持现有配置。

## 五、终端命令和日志

这一类测试里，终端日志比页面更重要。

```bash
# 先看驱动和模块
lsmod

# 看网络
ip addr
ip route
nmcli dev status

# 看日志
journalctl -f
dmesg -w

# 看总线
candump can0

# 诊断场景下常用的抓包思路
# 先确认会话，再确认响应，再确认复位后状态
```

如果是平台问题，我会优先看“终端是否发出去了”；如果是诊断问题，我会优先看“会话是否切对了”。

## 六、我怎么整理这部分经验

这部分内容最容易散，因为它横跨平台、诊断、网络、ECALL 和配置。

我后面把它整理成三个关键字：

1. **上报**
2. **联动**
3. **恢复**

只要这三个词跑通，测试思路就会比较清楚。

## 七、复盘

这段经历让我对“车载终端测试”有了更具体的理解：

- 不是单看某个功能是否成功。
- 而是要看功能之间是否互相影响。
- 不是只会点 UI。
- 而是要能从日志、总线和平台三个角度交叉验证。

如果说前一篇更偏底盘，这一篇更偏联动。两篇合起来，基本就能覆盖我在这段实习里最核心的测试工作。
