## 为什么需要楼层切面

这篇记录的是我在武汉市测绘研究院算法实习期间，做外立面缺陷检测三维系统时整理的一块报告可视化功能。

前面的流程已经能把外立面裂缝、破损、污渍等缺陷从二维照片定位到三维 OSGB 模型上，并在模型中显示红色标记。但如果最终交付只给一张三维截图，阅读成本还是偏高：

- 三维视角不同，缺陷点之间的相对位置容易被透视关系影响。
- 外立面模型有遮挡和立面转角，截图很难一次表达所有楼层。
- 报告读者不一定熟悉三维浏览器操作，更希望看到楼层级平面示意。
- 后续复查通常按楼层、立面区域和缺陷编号推进，需要稳定的索引图。

所以系统里需要一个更偏报告表达的视图：楼层横切面图。它不是为了替代三维模型，而是把三维定位结果压到一个更容易阅读的二维平面中。

## 切面图要解决什么问题

楼层切面图的目标很明确：每一层输出一张图，图里有该楼层的建筑轮廓、方向参考、缺陷点位置和缺陷编号。这样报告读者不用打开三维软件，也能快速判断缺陷大致在哪一层、哪一侧、离其它缺陷有多远。

我对切面图的要求主要有四个：

- **位置稳定**：同一个缺陷在重复导出报告时，应该落在同一张切面图的同一位置。
- **比例可信**：图中轮廓和缺陷点不能只是示意，至少要保持工程坐标下的相对位置关系。
- **信息克制**：报告图不能堆太多三维细节，只保留轮廓、缺陷点、编号和必要坐标。
- **能批量生成**：项目里缺陷可能很多，不能依赖人工一层层截图。

## 方案选择

最开始考虑过三种方案。

第一种是直接截取当前三维视角。这个做起来最快，但问题也最明显：相机位置、缩放比例和视角稍微变化，输出图就不一致。更麻烦的是，截图会包含纹理、阴影和透视变形，报告里很难稳定复现。

第二种是把 OSGB 模型整体投影成俯视图。这个方式能得到一个建筑平面轮廓，但缺陷属于不同高度，全部压到一张图里会丢失楼层信息。对于多楼层外立面缺陷，阅读体验不够好。

第三种是按楼层高度生成水平横切面。它从上往下看建筑轮廓，每一层只展示该高度范围内的缺陷点。这个方案最符合报告需求，所以项目中最终采用这一类思路。

项目中明确采用水平楼层切面，从上往下查看建筑轮廓，并把楼层范围限定在缺陷最低点到最高点之间。这样既能减少无效楼层，又能让报告篇幅跟实际缺陷分布匹配。

这个方案比直接使用当前视角截图更稳定，因为报告读者不需要理解三维视角和透视关系，只要看到某一层的轮廓和缺陷点位置即可。

## 输入数据

楼层切面图不是孤立功能，它依赖前面多个模块产出的结果。

| 输入 | 作用 |
| --- | --- |
| OSGB 三维模型 | 提供建筑整体轮廓和空间范围 |
| 缺陷三维坐标 | 决定每个缺陷属于哪一层、落在图上哪里 |
| 楼层高度规则 | 把连续的 Z 值拆成可读的楼层区间 |
| 坐标转换参数 | 处理 OSGB 局部坐标、工程坐标和绘图坐标之间的映射 |
| 缺陷编号和类别 | 用于在报告图中标注点位 |

其中最容易出问题的是坐标转换。OSGB 模型经常不是直接以真实工程坐标作为绘制坐标，浏览器或 OSG 加载时也可能引入局部原点、偏移量和轴向差异。如果切面轮廓和缺陷点使用了两套坐标系，图上就会出现“轮廓是对的，点偏到外面”的情况。

## 整体流程

楼层切面生成可以拆成五步。

### 1. 计算高度范围

如果直接从模型包围盒的最低点到最高点切，可能会生成很多没有缺陷的楼层图。报告里最需要的是缺陷出现的楼层，所以我优先根据缺陷点的 Z 值确定范围，再向上下各扩一点余量。

比如最低缺陷点在 14.2m，最高缺陷点在 36.8m，楼层高度按 3.6m 估算，那么只需要生成覆盖这些高度的切面图。无缺陷楼层可以不导出，或者只在统计页说明没有检测到缺陷。

### 2. 拆分楼层区间

楼层区间不一定非要等同于真实楼层编号。工程实现里可以先按统一层高切分，后续再根据项目资料修正为真实楼层。

这里我更关注两个边界：

- 缺陷点落在楼层边界附近时，不能因为浮点误差被分到错误楼层。
- 楼层区间要有稳定命名，否则报告重新导出后编号会变化。

因此实际处理时会给楼层边界加一个很小的容差，并把楼层的 `floor_id`、`z_min`、`z_max` 固定写入中间结果。

### 3. 获取楼层轮廓

理想情况下，可以直接对 OSGB 网格和水平平面求交，得到该高度处的外轮廓线。但 OSGB 通常是分块、分层级、带纹理的三维瓦片，直接在 Python 里处理全部三角面并不轻量。

因此工程里更现实的做法是：

- 优先使用切割/轮廓导出工具得到某一高度的轮廓结果。
- 在 Python 端读取轮廓点并做坐标归一化。
- 如果某一层轮廓缺失，就使用相邻楼层轮廓或主楼栋投影轮廓作为兜底。

这个兜底很重要。报告生成不应该因为某一层轮廓提取失败就完全中断，应该能提示“该楼层轮廓使用近似结果”，并继续生成其它内容。

### 4. 投影缺陷点

缺陷点本来是三维坐标 `(x, y, z)`。对于楼层切面图来说，Z 只用于判断属于哪一层，真正绘图时只保留 `(x, y)`。

投影逻辑大致是：

```text
for marker in markers:
  if floor.z_min <= marker.z < floor.z_max:
    point2d = worldXYToCanvas(marker.x, marker.y)
    drawMarker(point2d, marker.id, marker.category)
```

如果同一层里多个缺陷点距离很近，编号文字可能互相遮挡。我的处理思路是点位保持真实位置，文字标签可以做轻微避让；这样既不破坏点的位置，又能保证报告可读。

### 5. 绘制并写入报告资产

最终输出不是直接写入 Word，而是先生成切面图图片，再由报告模块统一引用。这样可以让报告模块只关心“插入哪张图”，不用重复处理三维坐标和绘图逻辑。

楼层切面图一般会包含：

- 楼层名称或高度范围
- 建筑外轮廓
- 方向箭头或坐标轴
- 缺陷点和缺陷编号
- 必要的图例和比例说明

缺图占位：这里后续可以补一张“某楼层切面图生成前后对比”，左侧为 OSGB 三维模型中的红点，右侧为报告中的楼层切面图。

## 楼层切面生成伪代码

```text
function generateFloorSections(model, markers):
  minZ, maxZ = verticalRange(markers)
  floors = splitRangeByFloorHeight(minZ, maxZ)
  for floor in floors:
    plane = horizontalPlane(z=floor.centerHeight)
    outline = intersectModelWithPlane(model, plane)
    floorMarkers = filter markers inside floor range
    image = drawOutlineWithMarkers(outline, floorMarkers)
    save image into report assets
```

伪代码只表达横切面思路。工程实现中会涉及 OSGB 局部坐标、工程坐标转换和轮廓导出工具。

## 坐标转换的坑

这个功能最容易踩坑的地方不是画图，而是坐标。

### 轮廓和红点不在一个坐标系

如果轮廓来自模型局部坐标，红点来自工程坐标，两者直接画在一起必然偏移。排查时不要只看数值大小，而要确认坐标来源：

- 轮廓点是否包含 `SRSOrigin` 或局部原点偏移。
- 红点是否已经做过空三坐标转换。
- X/Y 轴是否在某个工具中被交换。
- Z 轴单位是否和楼层高度单位一致。

我的经验是先找一个非常明确的点，比如楼栋转角或某个立面角点，把它在 OSGB、导出轮廓和最终画布中都标出来。只要这个锚点能对齐，后续缺陷点一般也能对齐。

### 图片坐标系和工程坐标系方向相反

工程坐标的 Y 轴通常向上，但图片画布的 Y 轴向下。如果没有翻转，切面图会出现上下镜像。这个问题看数值不一定明显，但一看方向箭头就能发现。

所以我在绘图阶段会显式处理坐标归一化：

```text
canvasX = (worldX - minX) * scale + padding
canvasY = (maxY - worldY) * scale + padding
```

这里用 `maxY - worldY`，就是为了把工程坐标转换成图片坐标。

### 楼层边界附近的缺陷归属

真实检测点的 Z 值可能有误差，尤其是由照片反投影到模型表面时，落点会受网格精度、射线相交和坐标转换影响。如果一个点刚好在楼层边界附近，简单用 `<=` 和 `<` 判断可能会不稳定。

处理方式是引入容差，并在报告中保留原始高度：

```text
if floor.z_min - eps <= marker.z < floor.z_max + eps:
  assign marker to floor
```

如果一个点同时落入两个楼层的容差范围，可以按离楼层中心更近的原则分配。

## 和报告模块的关系

楼层切面图最终服务于 Word 报告。报告里通常不会只放一张图，而是按楼层或缺陷区域插入多张图，并在缺陷明细表中引用对应切面。

我的设计是让切面模块输出结构化结果：

```text
floor_sections/
  floor_01.png
  floor_02.png
  floor_03.png
floor_sections.json
```

其中 `floor_sections.json` 记录每张图对应的楼层范围、缺陷编号列表和图片路径。报告模块读取这个文件后，就能把切面图和缺陷明细关联起来。

这样拆分之后，切面生成失败、图片缺失或报告导出失败都能单独排查，不会全部混在 Word 生成逻辑里。

## 复盘

楼层切面图看起来只是报告中的一张辅助图片，但它其实把三维定位、坐标转换、模型切割和报告导出串了起来。

这部分给我的最大经验是：三维结果要交付给非算法读者时，不能只追求“模型里能看到”，还要把结果转成稳定、低阅读成本、可复查的表达方式。对于建筑外立面缺陷检测这种工程项目，楼层切面图就是连接算法结果和实际复查流程的一层关键表达。
