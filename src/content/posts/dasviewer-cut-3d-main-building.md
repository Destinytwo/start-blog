## 为什么要切割三维模型

这篇记录的是我在武汉市测绘研究院算法实习期间参与的外立面缺陷检测科研项目中的三维模型预处理部分。

实景三维成果通常覆盖范围很大，里面不只有目标楼栋，还包括道路、绿化、周边建筑、地面设施等内容。

对于建筑外立面缺陷检测系统来说，直接加载完整模型会带来几个问题：

- 三维界面里无关场景太多，用户难以聚焦目标建筑。
- OSGB 数据量更大，加载和发布成本更高。
- 缺陷红点和楼层切面不容易围绕目标楼栋组织。
- 报告中展示的空间对象不够清楚。

因此我使用 DasViewer 对实景三维模型做预处理，把主要楼栋单独裁出来。

## 切割的目的不是简单变小

模型切割的直接结果是数据范围变小，但更重要的是工程语义更清楚。

系统关注的是某一栋建筑的外立面缺陷，那么三维模型、红点、楼层切面和报告都应该围绕这栋建筑组织。如果完整场景里混入大量无关建筑，后续统计和展示都会变复杂。

切割后的收益主要有：

- 三维展示更聚焦。
- 用户复查缺陷更高效。
- 红点显示范围更明确。
- 楼层切面生成更稳定。
- 发布包体积更可控。

## 切割流程

简化流程如下：

```text
打开完整 OSGB 场景
  -> 定位目标楼栋
  -> 选择裁剪范围
  -> 导出裁剪后的模型
  -> 重新加载验证
  -> 检查纹理和坐标
  -> 接入检测系统
```

### 切割模型校验伪代码

```text
function cutMainBuilding(fullScene):
  open fullScene in DasViewer
  select region around target building
  export cut model as new scene

  reload cut model
  verify textures are not missing
  verify tile index can be loaded
  verify SRSOrigin is still valid

  pick several surface points
  compare picked coordinates with expected building range

  approve model only after linkage test passes
```

这里的重点是：切完以后一定要重新加载和校验，不能只看视觉上裁剪成功。

## 切割后的检查项

### 1. 纹理是否完整

如果导出时漏掉纹理目录，模型可能能加载，但显示灰白或黑块。后续截图、报告和红点展示都会受影响。

### 2. 瓦片索引是否正常

OSGB 是瓦片结构，裁剪后需要确保主入口和瓦片目录仍然能被查看器识别。

### 3. 坐标原点是否一致

这是最重要的检查项之一。检测系统后续要做二维三维联动，如果模型裁剪后 SRSOrigin 或局部坐标发生异常，缺陷点会偏移。

### 4. 包围盒是否合理

可以计算裁剪后模型的 bounding box，检查范围是否只覆盖目标楼栋。这个范围后续也可用于楼层切面和点位过滤。

### 5. 与照片联动是否仍然正确

最终校验不是“模型能看”，而是“模型能和照片对上”。可以选几个明显点位，检查点击模型后候选照片是否合理。

## 模型切割和报告的关系

模型切割不仅影响三维界面，也影响报告输出。

报告中需要展示楼层切面和缺陷点分布。如果模型包含太多无关场景，切面图可能会出现道路、周边建筑等无关轮廓，降低可读性。

裁剪为主要楼栋后，报告里的空间表达会更清晰：

- 楼层轮廓只围绕目标建筑。
- 缺陷点分布更集中。
- 读者更容易理解缺陷位置。

缺图占位：这里后续可以补一张“完整场景切面 vs 主要楼栋切面”的对比图。

## 注意事项

在项目中我总结了几个注意点：

- 切割前先备份完整模型。
- 切割后不要手动乱改目录结构。
- 导出模型后立刻在系统里测试加载。
- 坐标校验要比视觉检查更重要。
- 如果后续需要和空三照片联动，必须保留坐标链路。

## 小结

DasViewer 切割模型是一个看起来很小、实际很关键的预处理步骤。

它让系统从“加载一个大场景”变成“围绕目标楼栋做缺陷检测和空间表达”。

对于建筑外立面缺陷检测项目来说，模型切割的价值在于：

```text
减少无关场景
  -> 聚焦目标建筑
  -> 稳定三维红点
  -> 改善楼层切面
  -> 提升报告可读性
```

这也是实景三维数据进入工程系统前必须处理的一环。
