## 写在前面

本文整理 **2026 年 3 月到 2026 年 7 月算法实习** 期间参与的建筑外立面缺陷检测科研项目。项目目标不是单独训练一个缺陷检测模型，而是围绕建筑外立面巡检场景，把无人机航拍影像、实景三维模型、目标检测、人工复核、空间定位和 Word 报告导出串成一个可演示、可复查、可交付的系统原型。

项目可以概括为一句话：

> 用无人机照片发现建筑外立面缺陷，再把二维检测结果回到三维模型中定位，最后形成可复核、可统计、可导出的巡检成果。

这类项目的难点并不只在算法指标。真正进入工程链路后，问题会变成：高分辨率图片如何检测小缺陷、OSGB 模型如何加载、二维检测框如何落到三维坐标、人工修正如何保留痕迹、报告导出如何不卡界面、发布包如何带齐运行时依赖。

![建筑物缺陷检测系统检测结果界面](/blog-assets/detection-ui-result.png)

> 检测结果界面：二维照片、检测框、类别统计和复核入口集中在同一个工作台中。

## 项目背景

传统建筑外立面巡检高度依赖人工观察和现场记录。对于高层建筑来说，人工方式存在几个明显问题：

- 安全风险高：外立面巡检涉及高空作业或靠近建筑边缘的观察。
- 数据复查困难：纸面记录或零散照片不容易回到具体空间位置。
- 结果交付成本高：人工整理缺陷、照片、位置和报告耗时较长。
- 历史对比困难：如果没有统一坐标和项目数据库，后续复查缺少基准。

无人机航拍可以降低采集成本，实景三维模型可以表达建筑空间结构，目标检测算法可以提高缺陷初筛效率。因此项目尝试把这三者结合起来，形成一个面向建筑外立面巡检的系统闭环。

项目中主要围绕以下方向推进：

- 缺陷检测：使用 SAHI + Ultralytics 处理大图小缺陷问题。
- 工程系统：使用 PyQt5 + FastAPI 构建桌面端和推理服务。
- 三维联动：基于空三参数、相机位姿和 OSGB 模型做二维三维关联。
- 人工复核：集成 LabelMe，支持漏检目标补标和结果回写。
- 报告导出：汇总唯一缺陷、楼层切面图、统计信息和 Word 报告。

## 系统整体流程

系统不是一次脚本推理，而是一套工程流程。一个项目从数据导入到报告导出，大致经历以下几个阶段：

```text
创建项目
  -> 导入 OSGB 实景三维模型
  -> 导入 UAV 航拍影像和空三成果
  -> 批量检测外立面缺陷
  -> 人工复核和补充漏检
  -> 二维缺陷框映射到三维模型
  -> 缺陷去重、统计和楼层切面生成
  -> 导出 Word 检测报告
```

如果只看模型推理，系统似乎很简单；但如果要让结果能被复查和交付，就必须保存完整上下文，包括项目目录、原始照片、检测标签、人工修改记录、三维点位、报告资产和运行日志。

## 技术架构

系统采用客户端和检测服务分离的架构。

### PyQt5 客户端

客户端负责与用户交互，主要功能包括：

- 项目管理：创建项目、打开项目、维护项目路径和资产目录。
- 图片管理：导入航拍照片、浏览原图、展示检测结果。
- 三维模型：加载 OSGB 实景三维模型，支持视角浏览和点位显示。
- 检测入口：批量提交图片到后端服务，并显示进度。
- 人工复核：打开 LabelMe 对检测结果进行修正。
- 报告导出：按项目记录生成 Word 报告和相关资产。

客户端的核心价值是把模型输出变成“可操作的工程结果”。检测框不是简单画在图上，而是要能被保存、筛选、修正、统计和关联到三维位置。

### FastAPI 检测服务

检测服务负责模型推理，主要处理：

- 接收客户端传入的图片路径或图片文件。
- 调用 SAHI 切片推理。
- 使用 Ultralytics 模型输出缺陷框。
- 将切片结果映射回原图坐标。
- 返回结构化标签数据，而不是只返回结果图片。

这种拆分有两个好处：

- 推理服务可以独立调试，便于记录耗时、显存和异常。
- 客户端不被模型依赖绑定，后续替换模型或切换推理策略更清晰。

### SQLite 项目数据库

数据库主要保存项目级状态：

- 项目信息和路径配置。
- 照片列表、检测状态和结果路径。
- 缺陷类别、置信度、来源和复核状态。
- 三维点坐标、楼层信息和去重后的唯一缺陷。
- 报告导出记录。

这个数据库不追求复杂，但必须稳定。桌面端项目最怕数据散落在多个目录里，后续无法判断哪个文件是当前项目的最新结果。

## 核心难点一：高分辨率外立面图片中的小缺陷

无人机拍摄的外立面图像往往分辨率很高，但裂缝、剥落、泛碱等缺陷在整张图中的比例很小。直接把整图缩放到 YOLO 输入尺寸后，小缺陷会被压缩到很少的像素，模型很难稳定识别。

因此项目中没有简单地“把图片丢给 YOLO”，而是引入 SAHI 切片推理：

- 把大图切成多个重叠小图。
- 在切片上进行目标检测。
- 将局部检测框映射回原图。
- 合并重叠区域中的重复框。

![高分辨率图片切片检测流程示意图](/blog-assets/sahi-tiling-workflow.svg)

> 大图切片检测流程：切片后保留更多缺陷细节，再把检测结果还原到原始图片坐标。

## 核心难点二：检测结果必须支持人工复核

建筑缺陷检测很难完全依赖模型一次性完成。现场图像里存在窗框、阴影、污渍、反光、贴片、空调外机等复杂背景，漏检和误检都无法完全避免。

所以系统必须支持人工复核。实现思路是把 LabelMe 集成到工作流中：

- 模型先输出初始检测框。
- 用户在系统中选择需要复核的图片。
- 打开 LabelMe 查看并修改标签。
- 系统读取修正后的标签并回写项目结果。
- 人工新增目标保留来源信息，便于后续分析漏检。

这里有一个重要设计：不要把人工复核当成“覆盖模型结果”的过程，而要把它当成“记录模型不足”的过程。模型漏检的目标后续可以沉淀为再训练样本。

![LabelMe 标注截图](/blog-assets/labelme-manual-annotation.png)

> LabelMe 复核界面：模型结果可以继续人工补充，保证检测结果最终可用。

## 核心难点三：二维检测框如何回到三维模型

如果检测结果只停留在二维照片里，报告里只能说“某张照片上有缺陷”。但建筑巡检更关心的是“缺陷在建筑物哪个位置”。

因此系统需要把二维检测框映射到三维模型上。简化来看，过程是：

```text
检测框中心点
  -> 像素坐标
  -> 相机坐标系射线
  -> 世界坐标系射线
  -> 与模型表面或近似平面求交
  -> 三维缺陷点
```

这个过程依赖空三成果中的相机内参、外参、照片位姿，以及 OSGB 模型坐标。实际工程里还要处理模型局部坐标、SRSOrigin、去畸变照片、坐标轴方向和多照片重复缺陷。

## 核心难点四：OSGB 模型不是普通 3D 文件

实景三维模型通常不是一个单独的模型文件，而是包含 metadata.xml、瓦片目录、纹理、坐标原点和大量 OSGB 文件的成果包。加载它时需要识别主索引、处理瓦片路径，并保证纹理和空间坐标都正确。

项目里还做了一个前置处理：用 DasViewer 对完整场景进行裁剪，只保留主要楼栋。这样做不是单纯为了让模型变小，而是为了让系统关注对象更清晰：

- 三维浏览时不会被周边道路和绿化干扰。
- 缺陷红点集中在目标楼栋上。
- 楼层切面和报告统计围绕同一建筑范围生成。
- 模型加载和发布包体积更可控。

![DasViewer 从完整场景切割主要楼栋的示意图](/blog-assets/dasviewer-cut-workflow.svg)

> DasViewer 切割流程：完整实景三维场景被裁剪为主要楼栋，服务后续定位和报告。

![DasViewer 切割前后对比截图](/blog-assets/dasviewer-cut-before-after.png)

> 模型切割前后对比：减少无关场景，让三维展示聚焦目标建筑。

## 报告导出：把系统结果变成可交付成果

检测系统如果只停留在界面展示，就还没有形成完整交付。最终用户往往需要的是报告：缺陷在哪里、数量多少、类别分布如何、对应哪些照片、楼层切面怎么展示。

报告模块负责把以下内容组织到 Word 中：

- 项目信息和检测概况。
- 缺陷类别统计。
- 去重后的唯一缺陷明细。
- 对应原始照片或检测结果图。
- 楼层横切面图。
- 导出时间和项目资产路径。

报告导出放在后台线程执行，避免主界面卡死。图片资产采用外部目录管理，避免 Word 文件体积过大。

![缺陷检测报告导出流程示意图](/blog-assets/report-flow.svg)

## 系统伪代码：从检测到报告

以下用一段伪代码把系统主流程串起来：

```text
function runFacadeInspection(project):
  load project database
  load osgb model and camera metadata

  for image in project.images:
    predictions = detectionService.detect(image)
    save raw predictions
    render preview image
    write predictions into database

  reviewedLabels = open manual review workflow
  update database with reviewed labels

  for defect in reviewedLabels:
    marker = project defect box to model coordinate
    merge marker if nearby duplicated defect exists
    save unique defect

  floorSections = generate floor section images(unique defects)
  export word report(project, unique defects, floorSections)
```

这段伪代码不是实际代码，而是表达整个系统闭环：检测、复核、定位、去重、切面、报告。

## 工程要点

算法项目进入工程场景后，评价标准会从“模型能不能跑”变成“结果能不能被使用”。

在这个系统里，模型只是其中一个环节。真正决定项目可用性的，是数据是否可信、标签是否能复核、坐标是否能对齐、三维点是否能解释、报告是否能交付、发布包是否能在别的电脑上跑起来。

作为项目展示时，更适合定义为一个 **计算机视觉 + 实景三维 + 桌面工程 + 报告交付** 的综合项目，而不是单纯的 YOLO 检测案例。
