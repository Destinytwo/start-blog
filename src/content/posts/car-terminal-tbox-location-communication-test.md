## 写在前面

本文是从总览里拆出来的细分笔记，重点只讲一件事：TBOX 的定位和通信到底怎么测。

这部分看起来像基础项，实际最容易暴露系统状态问题。因为它同时会碰到电源模式、缓存、报文时序、模组重启和平台上报。只要其中任一环不稳，后续的业务功能就会一起受影响。

## 一、先看测试目标

TBOX 定位与通信测试主要关注四件事：

1. 定位能不能起来。
2. 报文能不能按时发。
3. 异常后能不能恢复。
4. 长时间运行会不会漂。

对应到实际场景如下：

| 场景 | 关注点 |
| --- | --- |
| 休眠唤醒后定位 | 唤醒后能否正常定位 |
| 上电后定位 | 冷启动后首帧是否正常 |
| 长时间定位 | 连续运行是否失效 |
| 单独重启模组 | 重启后是否影响自身和上层链路 |
| 辅助定位 | 缓存和辅助定位文件是否正确 |
| 漂移检测 | 静态 / 动态轨迹是否一致 |
| 广播报文时序 | 首帧和周期报文是否满足要求 |

![TBOX 定位与通信测试流程](/blog-assets/tbox-location-communication-flow.svg)

> TBOX 定位与通信测试流程：先清缓存和辅助定位文件，再验证上电/唤醒、首次定位、报文时序和长稳漂移，避免把旧数据误判成真实定位成功。

## 二、为什么要先清缓存

这个项目里的关键点是：不要一上来就跑定位，要先确认缓存状态。

原因很简单：如果休眠前的定位缓存还在，唤醒后系统可能会优先拿到旧时间、旧位置，表面上看“已经定位成功”，实际上数据是不干净的。

所以正式验证前，先做两件事：

- 清掉定位缓存。
- 确认辅助定位文件状态。

这样能把“真定位问题”和“旧缓存误导”分开。

## 三、报文时序怎么看

广播报文这块，关注的是首帧和周期发送。

- 首帧是否在要求时限内发出。
- 首帧是不是指定类型。
- 周期报文是否稳定。
- 压力场景下会不会掉周期。

实测时需要一边看终端日志，一边看总线抓包。如果只看一个界面，很容易把“页面刷新晚”误判成“终端没发”。

```bash
# 终端侧常用命令
journalctl -f
dmesg -w

# CAN 抓包
candump can0
candump can1

# 发送示例报文
cansend can0 43B#0102030405060708
```

这里的重点不是命令本身，而是它们对应的判断顺序：

1. 先看终端有没有起来。
2. 再看报文有没有出总线。
3. 最后看平台有没有收到。

## 四、长稳和漂移

长时间运行测试本质上是在找“慢性故障”。

这类问题不会在第一小时暴露，可能要过很久才出现：

- 定位慢慢飘。
- 辅助定位越来越不稳定。
- 模组偶发重启。
- 某些状态值开始滞后。

静态漂移和动态漂移需要分开看：

- 静态看台架不动时，数据会不会跳。
- 动态看实车或模拟路线时，轨迹会不会和实际路线一致。

如果只拿某一次定位成功当结论，后面很容易踩坑；长稳测试的价值就在于把偶发问题提前暴露出来。

## 五、模组单独重启

模组单独重启是个很有价值的验证项。

因为它能逼出两类问题：

- 模组自己能不能恢复。
- 上层应用有没有被牵连。

重启前后分别检查：

- 定位是否恢复。
- 报文是否继续发。
- 上层状态有没有卡住。
- 平台有没有出现数据断层。

## 六、辅助定位文件

辅助定位是加速定位的关键，但它也会制造依赖。

如果文件存在，定位可能更快；如果文件丢失，系统应该自动退化到普通定位，或者重新下载，而不是卡死。

因此需要专门做一个“删文件后验证”的场景，确认系统在资源缺失时的表现是否合理。

## 七、终端命令速查

这部分是实际排障时最常用的一组命令。

```bash
# 模块和驱动
lsmod

# 看网络状态
ip addr
ip route
ping 8.8.8.8

# 看无线状态
nmcli dev status
nmcli dev wifi list

# 看 CAN 总线
candump can0

# 观察定位相关日志
journalctl -f
```

如果出现“页面显示正常，但定位不对”的情况，通常先看日志，再看总线，再看平台，最后才回到界面本身。

## 八、这部分最容易踩的坑

1. 休眠前没有清缓存，导致唤醒后数据不干净。
2. 只看终端成功，不看平台是否真的收到。
3. 长稳测试时间不够，没暴露慢性问题。
4. 模组重启后只检查了自己，没看上层联动。
5. 报文时序只看界面，没看总线抓包。

## 九、结尾

TBOX 定位与通信测试是整个车载测试链路的底盘。它测得稳，后面的平台上报、ECALL、诊断联动和异常恢复才有基础。

优先抓住这一块时，可以先看三个关键词：

- 缓存
- 时序
- 恢复

这三个词基本就能覆盖这部分的主要问题。
